## Remote procedure calls: goal

 * **Objective** : to make a distributed application program as similarly as possible to a non-distributed application
   * The RPC model aims to make communication conceptually the same as invoking a local procedure.
   ```mermaid
     sequenceDiagram
     box  Local  Procedure  Call
     participant  ProcessC
     end
     ProcessC  ->>+  ProcessC: proc1(arg1, arg2)
     box  Remote  Procedure  Call
     participant  ProcessA
     participant  ProcessB
     end
     ProcessA  ->>+  ProcessB: proc1(arg1, arg2)
     ProcessB  ->>-  ProcessA: <br/>
   ```
 * There are <u>**two main ingredients* *</u>:
    * Remote invocation code <u>as transparent as possible.</u>
    * Remote invocation code generation <u>as automated as possible.</u>
 * **Benefit**: the developer can focus more on what their application does rather than worrying about implementing a repetitive initial template.
   * The complexities of interactions between machines and address spaces are simplified.
   * A high-level interface for data movement and communication is provided.
   * Many of the errors that can arise from low-level communication/transmission interactions are simplified, freeing the developer from having to explicitly reimplement error handling in each program.


## (1/2) Remote invocation transparency

 * One of the ingredients is remote invocation transparency: the goal is to make remote invocation code as transparent as possible, so that it resembles a local function invocation.

* The following example shows how to achieve this transparency:
  * On the left are the elements of a non-distributed application with a local call to the sum function. 
  * On the right is the same application with a remote call to the sum function:
    * ```application.c``` and ```library.c``` remain. ```client_stub.c``` and ```server_stub.c``` are added.
    * The code in ```client_stub``` and ```server_stub``` hides the details of remote execution from the “main(...)” function.
    * The ```library``` component of the application is now in another process, and they communicate using POSIX queues in this example. <br><br>
      ![Code organization for remote invocation using POSIX queues](/materials/topic-rpc/rpc/ssdd_rpc_drawio_10a.svg)


## (2/2) Automatic generation

* The other ingredient is the most automated generation possible of the remote invocation code, i.e., that the server stub and client stub can be obtained automatically from the definition of the remote function interface (add, etc. for the previous example).

* In this way:
    * It is guaranteed that the generated code that serves as the initial template is based on a grammar that can be validated (thus ensuring that the code works).
    * It facilitates the generation of new distributed applications as well as the transformation of existing applications into distributed ones


## A brief history of RPCs
 
* RPC (Remote Procedure Call): calls to remote procedures
  * By Birrel and Nelson (1985) in “Implementing Procedure Calls”
* Hybrid:
  * Procedure calls
  * Message passing
* RPCs are at the core of many distributed systems
  * They reached their peak with DCE (Distributed Computing Environment)
  * They have evolved into remote method invocation (CORBA, RMI) and service invocation (Web Services)


## Structure of RPCs

* RPCs offer a simple interface for building distributed applications over TCP/IP
  
   ![Software stack with RPC and marshalling](/materials/topic-rpc/rpc/ssdd_rpc_drawio_10.svg)
  
* In general, the elements involved are:
   * An **active** client, which performs an RPC to the server
      * The client stub performs client/server communication, sending a request to the server stub
      * The client request sends a message containing at least the operation code and arguments.
   * A **passive** server, which performs the service and returns the result of the RPC to the client
      * The server_stub responds to the client's request
      * The response to the client sends a message containing at least the result of the invoked function.

      ![Client, client_stub, server_stub, and server](/materials/topic-rpc/rpc/ssdd_rpc_drawio_16.svg)

* Basic concepts:
     * A **network service** is a collection of one or more remote programs
     * A **remote program** implements one or more remote procedures
     * A **remote procedure**, its parameters, and its results are specified in a protocol specification file written in an interface description language (IDL)
     * A server can support more than one **version of a remote program**
        * This allows the server to be compatible with service updates
     * A **stub** (or proxy) is responsible for converting the parameters and results during the remote procedure, as well as sending and receiving these values via messages.
        * Stubs (client stub and server stub) are independent of the server implementation.
        * They are dependent on the service interface
        * They are generated **automatically** by the RPC software


## Developing with RPC
 
1. Define the service interface using an interface definition language (IDL):
   * Each operation must be defined, along with its input and output arguments and its results
     
   ### Example: sum.x
   ```c
     program SUM {
        version SUMAVER {
           int SUM ( int a, int b ) = 1;
        } = 1;
     } = 99;
   ```

2. Compile the IDL interface to generate the files responsible for client and server communications:
   * Common options are:
     * **-N** for procedures with multiple arguments.
      * **-M** for code that can be used by multiple threads.
   	  * **-a** to generate all files, including a template Makefile for compilation.
     
   ### Example: use of rpcgen
   ```bash
   rpcgen -N -M -a sum.x
   ```

     * Based on the description of the service interface in IDL, the rpcgen utility automatically generates both the files related to communication and the files with a template that the programmer must fill in with the implementation and use of the service:
       * Common files on the client and server side: **sum_xdr.c** and **sum.h**
       * On the server side: **sum_svc.c** and **sum_server.c** (the latter must be changed with the implementation of the service interface)
       * On the client side: **suma_clnt.c** and **suma_client.c** (the latter must be changed with the implementation of the client that will use the service)
   
       ![Files generated by rpcgen and files written by programmer](/materials/topic-rpc/rpc/ssdd_rpc_drawio_81.svg)
 
3. The service is implemented on the server side.
   * The content of the functions generated in *sumar_server.c* is changed as follows.

   ### Example: suma_server.c
   ```c
    #include "suma.h"

    bool_t suma_1_svc ( int a, int b, int *result,
                        struct svc_req *rqstp )
    {
        *result = a + b;  // code to be added
         return 1;        // 1 -> result ok
    }
    
    int add_1_freeresult ( SVCXPRT *transp,
                            xdrproc_t xdr_result, caddr_t result )
    {
        xdr_free (xdr_result, result);
        return 1;
    }
    ```

4. Implement the client that uses the interface on the client side.
    * Replace all the content generated in *sumar_client.c* with the following.
  	 * In the example given in sumar_cliente.c, for the *main* function, the *sumar_1* function is a local call, even though internally it involves the use of a remote service.

   ### Example: suma_client.c
   ```c
    #include "suma.h"

    char *host = NULL ;
    int sumar_1 ( int a, int b ) ;
    int main ( int argc, char *argv[] )
    {
       if (argc < 2) {
           printf ("Usage: %s <host>\n", argv[0]);
           exit (-1);
       }
       host = argv[1];
       int ret = add_1(1, 2);
       printf("Result of %d + %d is %d\n", 1, 2, ret);
       
       return 0 ;
    }

    int add_1 ( int a, int b )
    {
        CLIENT *clnt ;
        enum clnt_stat retval ;
        int result ;

        clnt = clnt_create (host, ADD, ADDUP, "tcp");
        if (clnt == NULL) {
            clnt_pcreateerror(host);
            exit (-1);
        }

        retval = sum_1(a, b, &result, clnt);
        if (retval != RPC_SUCCESS) {
            clnt_perror (clnt, "call failed");
       }

       clnt_destroy (clnt);
       
       return result ;
    }
    ```

 5. Compile and generate the executables on the client and server sides.

    ### Example: individual compilation
    ```bash
    gcc -g -D_REENTRANT -I/usr/include/tirpc  -o suma_xdr.o     -c suma_xdr.c
    gcc -g -D_REENTRANT -I/usr/include/tirpc  -o suma_clnt.o    -c suma_clnt.c
    gcc -g -D_REENTRANT -I/usr/include/tirpc  -o client_sum.o   -c client_sum.c
    gcc -g -D_REENTRANT -I/usr/include/tirpc  -o server_sum.o   -c server_sum.c
    gcc -g -D_REENTRANT -I/usr/include/tirpc  -o suma_server.o  -c suma_server.c
    gcc -g -D_REENTRANT  -o suma_server  suma_svc.o  suma_server.o suma_xdr.o -lnsl -lpthread -ltirpc
    gcc -g -D_REENTRANT  -o suma_client  suma_clnt.o suma_client.o suma_xdr.o -lnsl -lpthread -ltirpc
    ```

6. To run the program, first execute the server and then the client.
        
   ### Example: running the server 
   ```bash
   acaldero@docker1:~/sd$ ./sum_server &
   
   acaldero@docker1:~/sd$ rpcinfo  -p localhost
     program vers proto   port  service
      100000    4   tcp    111  portmapper
      100000    3   tcp    111  portmapper
      100000    2   tcp    111  portmapper
      100000    4   udp    111  portmapper
      100000    3   udp    111  portmapper
      100000    2   udp    111  portmapper
          99    1   udp  34654
          99    1   tcp  41745
    ```

   ### Example: running the client on the same machine using *localhost*:
   
   ```bash
   acaldero@docker1:~/sd$ ./suma_client  localhost
   Result of 1 + 2 is 3
   
   acaldero@docker1:~/sd$ kill -9 %1
   acaldero@docker1:~/sd$ sudo rpcinfo -d 99 1
   [1]+  Killed                  ./suma_server
   ```

## IDL Language

* An Interface Definition Language (IDL) allows you to specify an **interface**:
  * **Service name** used by clients and servers
  * **Procedure names**
  * **Procedure parameters**
       * Input
       * Output
  * **Data types** of the arguments.
  * May include communication options.

* The interface is shared by client and server.
  * Compilers can be designed so that clients and servers are written in different languages
    
    ![IDL, stubs, and client/server](/materials/topic-rpc/rpc/ssdd_rpc_drawio_36.svg)

* Programming with an RPC package:
  * The programmer must provide:
    * The **interface definition** (IDL)
      * Procedure names
      * Parameters that the client passes to the server 
      * Results that the server returns to the client
    * The **client code**
    * The **server code**
  * The IDL compiler generates:
    * The **stub** **of the client**
    * The **server stub**
    * Can generate an auxiliary library to perform *marshalling/unmarshalling* in the stubs


## Marshalling

 * Parameters are **marshalled** to resolve **problems** in data representation:
   * Server and client may run on machines with different architectures (number of bits, byte order, etc.)
   * It is not possible to send a pointer since the client and server have different address spaces.

 * Possible data representation schemes:
   * **XDR** (e**X**ternal * *D**ata **R**epresentation) is a standard that defines the representation of data types (**RFC 1832**)
   * **CDR** is the common data representation of CORBA.
   * **XML** (e**X**tensible **M**arkup **L**anguage) is a tag-based metalanguage defined by W3C
   * **JSON** (* *J**ava**S**cript **O**bject **N**otation) is a lightweight text format for data exchange.

 * Example:  ‘Smith’, ‘Paris’, 1934

   ![Comparison between XDR, CDR, XML, and JSON](/materials/topic-rpc/rpc/ssdd_rpc_drawio_40.svg)


## Localization and binding

* **Binding service**: service that allows the association between the client and the server to be established:
   * It involves **locating the server** that offers a certain service
   * The server must **register its address** in a name service managed by a dynamic binding server (*binder*)
 
* The **dynamic binder** is the **service** that maintains a **translation table** between **service names** and **addresses**.  It includes functions for:
    * Registering a service name
    * Deleting a service name
    * Search for the address corresponding to a service name
* How to **locate** the dynamic binder:
   1. Run at a fixed address on a fixed computer
   2. The operating system is responsible for indicating its address
   3. Broadcast a message when processes begin execution.

* The steps for registration and binding are:
   1. Server obtains address
   2. Registers address
   3. Searches for server
   4. Returns address
   5. Request
   6. Response
   7. Deletes address (end of service)
     
   ```mermaid
   flowchart 
       subgraph id1 [ ]
       A["Client"]
       B["Server"]
       C["Binder"]
       end
       B --->|"(2)"| C
       B --->|"(7)"| C
       
       A --->|"(3)"| C
       C --->|"(4)"| A
       A --->|"(5)"| B
       B --->|"(6)"| A
       B -.->|"(1)"| B
   ```


## Types of errors that may occur with RPCs
   
* The client is unable to locate the server
  * Possible solutions:
    * Check if the server is down
    * Check that an old version of the server is not being used
* Message loss
  * The client's request message to the server is lost
    * Possible solution:
      * A timer is activated after sending the message and **if there is no response, the message is retransmitted**.
  * The server's response message to the client is lost
    * Possible solutions:
      * If it is an **idempotent** operation (it can be repeated and returns the same result), there is no problem.
      * With **non-idempotent** operations, requests that have already been executed must be discarded: **a sequence number and a field to indicate whether it is original or a retransmission** must be added * to the client and save a history of previous requests and their responses so that they are not executed again on the server.
* The server fails after receiving a request
  * Possible solutions:
    * **Do not guarantee anything**
    * **At least once** semantics: retry and ensure that the RPC is performed at least once (be careful with non-idempotent operations)
    * **At most once** semantics: do not retry, it may not be performed or used only once
    * **Exactly once** semantics
* The client fails after sending a request
  * Possible solution:
    * Have the ability to cancel an ongoing operation.


## Characteristics that define an RPC

* Types of IDL:
   * Integrated with a programming language (Cedar, Argus, Java RMI)
   * Specific interface definition language to describe the interfaces between clients and servers (Sun/ONC RPC, DCE RPC, CORBA, Google RPC)

* Parameter types:
  * Input parameter (in)
    * The parameter is sent from the client to the server
  * Output parameter (out)
    * The parameter is sent from the server to the client
  * Input/output parameter (inout)
    * The parameter is sent both from the client to the server and then from the server to the client

* RPC models
   * Synchronous RPC: request and response
   * Asynchronous RPC: request only
     * The client does not wait for the response
     * Does not support output parameters
     * Example in CORBA: *oneway* methods

* Possible data representation schemes:
  * **XDR** (e**X**ternal **D**ata **R**epresentation)
  * **CDR** is the common data representation of CORBA.
  * **XML** (e**X**tensible **M**arkup **L**anguage) is a tag-based metalanguage defined by W3C
  * **JSON** (**JavaScript Object Notation) lightweight text format for data exchange.

* Types of **data conversion**:
  * **Asymmetric** data conversion:
    * The client converts to the server's data representation
  * **Symmetric** data conversion:
    * The client and server convert to a standard data representation
    * ONC-RPC uses symmetric data representation
 
* Types of **typing**:
    * **Explicit** typing:
      * Each piece of data sent includes the value and its type
    * **Implicit** typing:
      * Each piece of data sent only includes the value, the type is not included
      * ONC-RPC uses implicit typing

* **Connection** types:
  * **Non-persistent** connection: the connection between client and server **is established for each RPC**
    * More fault-tolerant
    * Allows service migration
  * **Persistent** link: the link **is maintained after the first RPC**
     *  Better performance in applications with many repeated RPCs
     * Problems if servers change location
  * **Hybrid** models

 * Types of error indications:
   * Return an **error code** (-1 / NULL) or the output value
     * To be avoided: the error code is within the range of possible output values, such as -1 for ```sum(a,b)```
   * **Raise an exception**
      * Requires a language that has exceptions
   * Return **two values**: the **status of the operation** (OK / KO) and the **output value**
     * Better, although it reduces transparency: ```int res = add(a,b, &status)```
   * Return ** a pointer to the value**: 
      * If the pointer is NULL, there is an error and no valid value.
      * If the pointer is not NULL, then accessing that pointer gives the desired value.
      * For multi-threaded environments, the pointer may point to dynamic memory that must be freed after use.

* Failure strategies:
    * RPC client
      * Resend request after timer
      * Include request identifier (sequence number)
    * RPC server
      * Analyze sequence number to filter duplicate requests (save history and do not re-execute repeated requests)
      * Offer semantics:
        * **Do not guarantee anything**
        * **At least once**: retry and guarantee that the RPC is performed at least once (be careful with non-idempotent operations)
        * **At most once**: do not retry, may not be performed or used only once
        * **Exactly once**

* Types of protocols used:
  * Is the protocol connection-oriented?
    * Connection-oriented
    * Not connection-oriented
  * Is the protocol standardized?
    * Standard (e.g., TCP or UDP)
    * Specific (non-standard)


## Sun/ONC RPC

* Brief history:
   * Initially designed for the NFS file system
   * Described in RFC 1831
   * Also called ONC-RPC (*Open network computing*)

* Features:
  * Both UDP and TCP can be used
      * Client and server must agree on the transport protocol to be used
  * **At least once semantics**
  * **XDR** interface definition language
  * **Implicit typing**
  * **Symmetric data representation**
  * Includes:
     * A stub generator called rpcgen
     * RPC support function libraries (<rpc/rpc.h>)
     * Procedure registration service (*portmapper* or *rpcbind*)
 
* Support: 
   * For C on UNIX/Linux, Windows, etc.
   * For Java
       * http://sourceforge.net/projects/remotetea/


## XDR for Sun/ONC

 * XDR (eXternal Data Representation) is a standard that defines the representation of data types (RFC 1832)
   * Initially used for external data representations
   * Extended to interface definition languages

* Main features of XDR:
  * Uses big-endian representation
  * Does not use negotiation protocol
  * All elements in XDR are multiples of 4 bytes
  * Message transmission uses implicit typing (the data type does not travel with the value)
  * Uses symmetric data conversion
  * Data is encoded in a byte stream
  * Memory management: RPCs do not reserve dynamic memory.
    * If input and/or output arguments use dynamic memory, then:
      * The server may have to reserve memory and implement the corresponding *xxx_freeresult(...)*.
      * The client may have to reserve memory before invoking the RPC and then free it with *xdr_free(...)*


## IDL: base format

#### Interface definition examples:
 * With one input parameter and one output parameter:
   ```c
   struct args {
      int a;
      int b;
   };
   struct res {
          int r;
   };
   program ADD {
      version ADDVER {
         struct res ADD       (struct args a) = 1;  // 1 is the procedure number
         struct res SUBSTRACT (struct args a) = 2;
      } = 1;    // 1 is the version number
   } = 99;      // 99 is the program number
   ```
 * With multiple arguments (requires rpcgen with at least the -N option):
   ```c
   program ADD {
      version ADDVER {
          int ADD      (int a, int b) = 1;    // 1 is the procedure number
          int SUBTRACT (int a, int b) = 2;
      } = 1;    // 1 is the version number
   } = 99;      // 99 is the program number
   ```


#### An interface definition can contain:
* <u>Data type definitions at the beginning of the file</u>:
   ```c
     struct res {
         int status;
         int value;
     };
     ...
   ```
* <u>A set of remote procedures (following any type definitions)</u>:
   ```c
     ...
     program ADD {
         version ADDVER {
            struct res ADD  (int a, int b) = 1;  
         } = 1;    
     } = 99;      
   ```
   * **Each remote procedure is uniquely identified by three fields** encoded with unsigned integers:  (NUM-PROG, NUM-VERSION, NUM-PROCEDURE)
     * NUM-PROG is the remote program number.
       * They are defined in RFC 1831 (http://www.ietf.org/rfc/ rfc1831.txt)
         * ```0x00000000 - 0x1fffffff``` --> defined by Sun 
         * ```0x20000000 - 0x3fffffff``` --> defined by user
         * ```0x40000000 - 0x5fffffff``` --> transient
         * ```0x60000000 - 0xffffffff``` --> reserved
       * **For laboratories, it is recommended to use the NIA of one of the group members to avoid interfering with the services of other groups.* *
     * NUM-VERSION is the program version number.
       * The first implementation (version) must be 1.
       * Each time a change is made to the IDL interface (procedures are added/removed/modified), the version is incremented.
     * NUM-PROCEDURE is the remote procedure number.
       * These are specified by the programmer and must be different for each procedure.
   * **Input parameters**:
      * By default (without using the -N option in rpcgen), procedures only accept one input parameter. 
      * If there is more than one parameter, it is possible to either **encapsulate all the parameters in a structure and pass that structure** or **use the -N option in rpcgen**.
      * Passing a pointer does not make sense since it points to the local address space and to an arbitrary area of data in remote space. To solve this problem, an RPC system can:
         * Prohibit the use of pointers as arguments.
         * Allow the use of pointers, but ensure that what is transmitted is the data referenced by the pointer and not the pointer itself.
      * **It may not be possible to pass a fixed-size array/vector**, 
        for example:
        ```c
        int invalid_routine ( int arg[32] );
        ```
        **but it may be possible to use a structure containing a field that is a fixed-size array/vector** instead,
        for example:
        ```c
        struct arr_double_struct { double vector[32]; } ;
        int valid_routine ( struct arr_double arg );
        ```
   * **Output parameters**:
       * The function returns a single result (of the type indicated to the left of the function name).
       * If more than one result needs to be returned, **it is possible to define a structure containing all the values to be returned and return that structure**.
   * **Input and output parameters**:
       * It is common for there to be NO input and then output parameters at the same time in the IDL definition.
       * **To simulate an input and output parameter**, two parameters must be used in the IDL definition: one input (as an additional input parameter) and one output (which joins the rest of the output parameters that the function has to return).

## The rpcbind/portmapper process

* The link in ONC-RPC is made using a process called rpcbind (portmapper).
  * Supports TCP and UDP
 
* Basic operation: 
  * An rpcbind process runs on each computer on a well-known port (111)
  * When a server starts up, it registers the service with the local rpcbind on the computer where it is running.
  * The rpcbind stores the following for each local service:
    * The program number
    * The version number
    * The port number
  * When a client needs to invoke a remote procedure, it sends the following to the rpcbind on the remote host (it needs to know the server's IP address):
    * The program number and version number
  * The rpcbind process returns the server port on that machine.


 * Dynamic linking:
   * The number of available ports is limited, and the number of potential remote programs can be very large
   * Only rpcbind will run on a specific port (111), and the port numbers where servers listen are found by querying rpcbind
   * The ```rpcinfo``` command can be used to query registered services:
      ```bash
     acaldero@docker1:~$ rpcinfo -p localhost
     program vers proto   port  service
     100000    4   tcp    111  portmapper
     100000    3   tcp    111  portmapper
     ...
     100005    1   udp  42264  mountd
     100005    1   tcp  48921  mountd
     100005    2   udp  33964  mountd
     ...
     100021    4   udp  47609  nlockmgr
     100021    1   tcp  42741  nlockmgr
     100021    3   tcp  42741  nlockmgr
     100021    4   tcp  42741  nlockmgr
     99        1   udp  46936
     99        1   tcp  40427
     ```


## Function library rpc.h

* Create a handler for the client:
  ```c
   CLIENT * clnt_create ( const char *host,
                           const u_long prognum,
                           const u_long versnum,
                           const char *nettype )
  ```
  * Arguments:
    * **host**: name of the remote host where the remote program is located
    * **prognum**: program number of the remote program
    * **versnum**: version number of the remote program
    * **nettype**: transport protocol: NETPATH, VISIBLE, CIRCUIT_V, DATAGRAM_V, CIRCUIT_N, DATAGRAM_N, TCP, UDP

 * Destroy the client handler:
   ```c
   void clnt_destroy ( CLIENT *clnt ) ;
   ```
   * Arguments:
     * **clnt**: client RPC handler (previously created with clnt_create)

* Indicate the error in an RPC failure:
  ```c
  void clnt_perror       ( CLIENT *clnt, char *s ) ;
  void clnt_pcreateerror ( CLIENT *clnt, char *s ) ;
  ```
  * Arguments:
    * **clnt**: client RPC handler
    * **s**: error message


## Notation used in code generated by rpcgen

* Based on the PROCEDURE definition in the .x file, for example:
  ```c
  struct res {
     int a ;
     int b ;
  } ;

  program PROGRAM {
     version PROGRAM_VER {
         struct res PROCEDURE ( int a, char b ) = 1 ;
     } = 1 ;
  } = 0x1001010 ;
  ```

* When executing **rpcgen with the multiple arguments option** (usually -N), the following is generated:

  * On the **server** side, prototype of a remote procedure to be implemented:
    ```c
    bool_t procedure_1_svc ( int arg1,
                                 char arg2,
                                 struct res *result,
                                 struct svc_req *rqstp );
    
    ```
    * Where:
      * Name of the function to be implemented: **procedure_1_svc**
        * starts with PROCEDURE...
        * ...everything is converted to lowercase...
         * ...followed by “_” , followed by the version (1 in this example) and followed by “_svc”
      * Arguments: **int a, int b, struct res \*result, struct svc_req \*rqstp**
        * arguments in the IDL
        * if the function returns a value, then:
          * pointer to the data type of the IDL return value
        * extra parameter of type “struct svc_req *rstp” containing request information
      * Return value: **boot_t**
        * 1 (TRUE) if successful and 0 (FALSE) to indicate a temporary error

* On the **client** side, prototype of a remote procedure to be adapted:
  ```c
      enum clnt_stat procedure_1 ( int a,
                                   char b,
                                   struct res *res,
                                   CLIENT *clnt );
  ```
  * Where:
    * Function name: **procedure_1**
       * starts with PROCEDURE...
       * ...everything is converted to lowercase...
       * ...followed by “_” followed by the version (1 in this example)
    * Arguments: **int a, char b, struct res \*ret, CLIENT \*clnt**
       * arguments in the IDL
       * if function returns a value then:
          * pointer to the data type of the IDL return value
       * extra parameter of type “CLIENT *” representing the handler of an RPC client
    * Return value: **enum clnt_stat**
       * RPC_SUCCESS if successful

## Authentication

* Request and response messages have fields for passing authentication information
* The server is responsible for controlling access
* There are different types of authentication protocols:
  * Null authentication (AUTH_NULL)
  * UNIX-style, based on uid and gid (AUTH_UNIX)
  * Kerberos authentication
  * Using a shared key that is used to sign RPC messages

Example skeleton with UNIX authentication:
   ```c
   CLIENT *cl = client_create("host", <SOMEPROG>, <SOMEVERS>, "udp");
   if (cl != NULL) {
       /* To set UNIX style authentication */
       cl->cl_auth = authunix_create_default();
   }

   /* use the authentication */
   ...

   /* destroy the authentication */
   auth_destroy(clnt->cl_auth);
   ```


## RPC in Python

* Installing *rpyc*:
   * Installation on the system:
     ```bash
     pip3 install rpyc
     ```
   * Installation for a user:
     ```bash
     pip3 install rpyc --user
     ```

* (1/2) Execution using *Classic RPyC*
  * Running a classic server that can execute functions and evaluate expressions in Python:
    ```bash
    /usr/local/bin/rpyc_classic.py
    ```
  * Client execution:
    * With eval + execute:
      ### client-1.python
      ```python
      import rpyc
      conn = rpyc.classic.connect("localhost")
      conn.execute('import math')
      x = conn.eval('2*math.pi')
      print(x)
      ```
    * With teleport:
      ### client-2.python
      ```python
      import rpyc
      conn = rpyc.classic.connect("localhost")
      def square(x):
             return x**2
      fn = conn.teleport(square)
      print(fn(2))
      ```

* (2/2) Execution using *RPC*
  * Calculation server example:
    ### calc_service.py
    ```python
         import rpyc
         from rpyc.utils.server import ThreadedServer

         class CalculatorService(rpyc.Service):
            def exposed_add(self, a, b):
                return a + b
            def exposed_sub(self, a, b):
                return a - b
            def exposed_mul(self, a, b):
                return a * b
            def exposed_div(self, a, b):
                return a / b

         if __name__ == "__main__":
            server = ThreadedServer(CalculatorService, port = 12345)
            server.start()
    ```
  * Example of a calculator service client:
    ### calc_client.py
    ```python
         import rpyc
         conn = rpyc.connect("localhost", 12345)

         x = conn.root.add(4,7)
         print(x)

         x = conn.root.sub(4,7)
         print(x)
    ```



## Using RPCs in the Ubuntu Linux distribution

There are three details to check in your Ubuntu installation:

1) The following packages must be installed: libtirpc, rpcsvc-proto, and rpcbind. <br> For the Ubuntu Linux distribution, run:
   ```bash
   sudo apt-get install  libtirpc-common libtirpc-dev libtirpc3 rpcsvc-proto rpcbind
   ```
2) You must start rpcbind.
   If there is an error when starting, please try the following:
   ```bash
   sudo mkdir -p /run/sendsigs.omit.d/
   sudo /etc/init.d/rpcbind restart
   ```
3) You may need to edit the **Makefile** and check that it includes the following snippet:
   ```make
   ...
   CFLAGS += -g -I/usr/include/tirpc
   LDLIBS += -lnsl -lpthread -ldl -ltirpc
   ...
   ```
